Doctrine

GGTC.info HTML Interface Doctrine V000

⸻

Doctrine 01 — Visible Content First

When building WordPress HTML, visible content must come first.

Do not create schema-only or metadata-only output when a visible page is requested.

A user-facing page must display readable content on the page.

⸻

Doctrine 02 — Flow Over Stacked Modules

Do not stack unrelated modules without pathway logic.

A GGTC.info page must move the user through a structured sequence.

The structure must answer:

* Where am I?
* What is this?
* What can I do?
* Where do I go next?
* How do I return?

⸻

Doctrine 03 — Interaction Must Support Understanding

Interaction must not exist only for visual effect.

Interaction must help users:

* learn
* choose
* move
* understand
* return
* continue

Buttons, cards, clocks, and panels must support ecosystem movement.

⸻

Doctrine 04 — The Clock Represents Continuity

The world clock layer must reinforce:

* global synchronization
* system presence
* publishing continuity
* infrastructure identity
* operational awareness

The clock is part of the ecosystem interface doctrine.

⸻

Doctrine 05 — Mobile-First Design

GGTC.info HTML must work on mobile devices.

Required mobile qualities:

* readable text
* rounded sections
* responsive grids
* buttons large enough to tap
* no unreadable dense panels
* clean vertical flow
* clear hierarchy

Mobile use is not secondary.

Mobile use is part of the production workflow.

⸻

Doctrine 06 — WordPress Compatibility

HTML must be suitable for WordPress Custom HTML blocks.

The HTML should:

* use scoped CSS
* avoid global style conflicts when possible
* keep JavaScript simple
* avoid external dependencies unless needed
* display correctly without theme-specific assumptions
* separate visible content from schema when schema is used

⸻

Doctrine 07 — Ecosystem Identity Must Repeat

The interface must preserve recurring identity markers.

Examples:

* GGTC.info name
* gold G identity
* rounded interface cards
* dark/gold/cyan visual language
* world clock structures
* approved social channels
* domain ecosystem references
* system line

Repetition creates recognition.

Recognition creates continuity.

⸻

Doctrine 08 — Loops Teach Users

The ecosystem uses loops to help users learn navigation behavior.

A good loop helps users:

* return
* continue
* recognize pathways
* understand system structure
* revisit key layers
* move through the ecosystem naturally

Loops are not accidental.

Loops are part of the learning architecture.

⸻

Doctrine 09 — Friction Must Be Intentional

Allowed friction:

stop and learn.

Not allowed friction:

confusion, unclear navigation, unreadable layout, broken pathway, or missing return logic.

Friction should serve education or orientation.

⸻

Doctrine 10 — HTML Is Infrastructure

For GGTC.info, HTML is infrastructure.

It supports:

* publishing
* navigation
* training
* global synchronization
* social continuity
* repository documentation
* ecosystem visibility
* user interaction

The page is not only content.

The page is a working system surface.
